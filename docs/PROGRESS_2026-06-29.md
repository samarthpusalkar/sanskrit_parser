### 3. Built the Sanskrit DSL Package (`sanskrit_dsl/`)
- `types.py` — `SutraSpec`, `SutraContext`, `SutraOperation`, `CompiledSutra` with real pratyahara matching and phonological computation (guna, vrddhi, dirgha, bijection)
- `parser.py` — Clean vibhakti parser that does NOT depend on the deprecated `SutraAstBuilder`. Prefers LLM-extracted metadata. Falls back to honest vibhakti parsing.
- `compiler.py` — Compiles sutras into executable `CompiledSutra` objects with anuvṛtti tracking
- `meta_engine.py` — `ParibhashaRegistry`, `AsiddhatvaEnforcer`, `AnuvrttiTracker`, `AntarangaResolver` with specificity-based conflict resolution
- `executor.py` — Real execution engine: finds best matching rule, resolves conflicts, applies operation, records trace evidence
- `corrections.py` — DEPRECATED (was hardcoded shortcuts, now does nothing)

### 4. Built an LLM Extraction Pipeline
- `tools/llm_sutra_extractor.py` — Sends sutras to Ollama (`deepseek-v4-pro:cloud`) with:
  - Commentary context (Vasu English translation)
  - Anuvṛtti context (previous 3 sutras)
  - Tripāḍī handling (special note for 8.2–8.4)
  - HTTP API (localhost:11434), JSON format, caching in DB

### 5. Deprecated the Old Broken Parser
- `compiler/ast_builder.py` `SutraAstBuilder` — misassigned vibhakti roles for many sutras
- `sanskrit_dsl/parser.py` now uses a clean parser that does NOT call `SutraAstBuilder`
- The clean parser is honest: it parses what it can, records hurdles for the rest

### 6. Wired the DSL Executor into the Benchmark Adapter
- `benchmarks/local_engine_adapter.py` now routes sandhi cases through `DSLExecutor`
- No longer uses the old `UniversalRuleEngine.dispatch_forward` path
